# Vision & Scope: Ревизия после Phase 0

> Date: 2026-03-18 | Status: DRAFT | Author: Artem + Claude Code

---

## 1. Что показала Phase 0

Phase 0 реализовала `.bobr/` формат + CLI для бэклога с зависимостями. Технически — всё работает: hash IDs, YAML frontmatter, `ready`/`claim`, SQLite cache, worktree workflow.

**Но обнаружилось фундаментальное противоречие:**

Я сам не пользуюсь Bobr CLI для управления своими задачами. Мне проще думать в Notion или просто в голове. CLI — это инструмент для агентов, а не для людей. А люди — это те, кто решает *что* делать и *зачем*.

### Корень проблемы

**У меня нет единого инструмента для управления требованиями и задачами, которым могли бы пользоваться и люди, и агенты.**

- **Люди** хотят Notion-like experience: быстрый ввод, визуальная навигация, drag-and-drop, все записи в одном месте
- **Агенты** хотят простые файлы на локальной машине: YAML/Markdown, CLI с JSON output, детерминированная структура

Это не две разные потребности — это одна потребность с двумя интерфейсами.

### Что PRD v0.8 делал не так

1. **Позиционирование через модели рынка** — вместо "решение конкретной боли" было "комбинация лучших черт трёх моделей". Красиво, но бессмысленно как компас для разработки.
2. **Слишком много концепций** — requirements (Вигерс), specs (OpenSpec), changes, knowledge base, consumer interfaces (Cowork, Telegram), Runner — каждая сама по себе продукт.
3. **CLI-first как стратегия** — удобно для агентов, неудобно для людей. Dogfooding проваливается: если автор не пользуется — никто не будет.
4. **"Tool Provider, not AI Consumer"** — правильный принцип, но доведённый до абсурда. Zero AI в ядре = Zero intelligence в UX. Нет помощи человеку.

## 2. Пересмотренное видение

### 2.1 Одно предложение

**Bobr — единый source of truth для требований, задач и знаний проекта, где люди работают через удобный интерфейс, а агенты — через файлы и CLI.**

### 2.2 Ключевой инсайт

Данные живут в `.bobr/` (git-native, портабельные, machine-readable). Но `.bobr/` — это *формат хранения*, а не *интерфейс*. Интерфейсов несколько:

```
                    ┌──────────────┐
                    │   .bobr/     │  ← единый source of truth
                    │  (git repo)  │     YAML + Markdown файлы
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         ┌────▼────┐  ┌───▼───┐  ┌────▼────┐
         │  Web UI │  │  CLI  │  │  MCP    │
         │(Notion- │  │(bobr) │  │ Server  │
         │  like)  │  │       │  │         │
         └────┬────┘  └───┬───┘  └────┬────┘
              │            │            │
          Люди         Оба          Агенты
```

### 2.3 Что Bobr делает

1. **Хранит** требования, задачи и знания в `.bobr/` — git-native, портабельно, версионируемо
2. **Предоставляет интерфейсы** — web UI для людей, CLI для обоих, MCP для агентов
3. **Связывает** бизнес-потребность → задача → код → результат (traceability)

### 2.4 Чем Bobr НЕ является

- **Не замена IDE** — не пишет код, не запускает агентов
- **Не AI-продукт** — не содержит встроенного AI (но AI-агенты — первоклассные клиенты)
- **Не Jira/Linear** — не пытается повторить enterprise PM, фокус на малых командах с AI

## 3. Пересмотренные принципы

### Сохраняем (проверены Phase 0)

1. **Git-native storage** — `.bobr/` в репозитории, YAML+MD, hash IDs
2. **Structured data over natural language** — метаданные как YAML, а не свободный текст
3. **Dependency graph** — `ready`/`claim`/`blocked` — это работает
4. **Tool Provider** — Bobr предоставляет данные, не вызывает AI API

### Меняем

| Было | Стало | Почему |
|------|-------|--------|
| CLI-first | **Interface-plural** | Люди не пользуются CLI для управления задачами |
| Zero AI в ядре | **AI-assisted где полезно** | "Tool Provider" не исключает умный UX — генерация описаний, приоритизация, декомпозиция |
| Requirements по Вигерсу + Knowledge Base + Specs + Changes + Runner | **Tasks + Requirements + Knowledge** | Слишком много концепций. Specs и Changes — это workflow поверх задач, а не отдельные сущности |
| "CEO нажимает Запустить" | **"Все видят одну картину проекта"** | Runner (запуск агентов) — это не наша проблема. Наша проблема — общий контекст |
| Догфудинг через CLI | **Догфудинг через web UI** | Если я сам хочу Notion — значит мне нужен web-интерфейс для Bobr |

### Убираем (из ближайшего scope)

- **Bobr Runner** (запуск агентов) — не наша проблема, Claude Code/Codex делают это сами
- **Consumer interfaces** (Telegram, Cowork) — это интеграции, а не ядро
- **Token budget as capacity** — преждевременная оптимизация
- **Wave planner** (параллельные агенты) — нужен когда есть хотя бы один рабочий workflow
- **11-phase feature workflow** — overengineered; простой workflow: idea → task → PR → done

## 4. Пересмотренный scope (что дальше)

### Фокус: три слоя

**Слой 1: Формат (уже есть)**
- `.bobr/` структура, YAML+MD, hash IDs, SQLite cache
- CLI: `bobr init`, `backlog add/list/edit`, `dep add/remove`, `ready`, `claim`

**Слой 2: Удобство для людей (нужно сделать)**
- Web UI для просмотра и управления бэклогом (Notion-like)
- Быстрое добавление задач, drag-and-drop приоритизация
- Визуализация зависимостей
- Requirements и knowledge — как страницы, а не CLI-команды

**Слой 3: Интеграция с агентами (нужно сделать)**
- MCP Server (агент подключается к Bobr как tool provider)
- `bobr context generate` — агент получает structured контекст для работы
- CLAUDE.md / AGENTS.md автогенерация

### Что сделать первым

Главный вопрос: **где я буду вести бэклог Bobr?**

Если ответ "в Notion, потому что web UI ещё нет" — значит Bobr не решает свою основную задачу. Значит следующий шаг — минимальный web UI, чтобы я сам перешёл на Bobr.

## 5. Открытые вопросы

1. **Web UI сразу или Notion sync?** — Строить свой web UI дорого. Альтернатива: двусторонний sync `.bobr/` ↔ Notion. Но это хрупко и создаёт зависимость.

2. **Scope requirements management** — Нужен ли полный Вигерс (Vision & Scope, Use Cases, Business Rules)? Или достаточно простых "goals" и "user stories"?

3. **Knowledge base scope** — Полноценная KB с семантическим поиском? Или просто директория `.bobr/knowledge/` с Markdown-файлами?

4. **Solo-first или team-first?** — Phase 0 показала: даже для solo-разработчика с агентами инструмент должен быть удобным. Стоит ли сфокусироваться на solo use case, прежде чем масштабировать на команды?
