## PRESS RELEASE

### Bobr: Open-source платформа, которая дала AI-агентам контекст, а командам — прозрачность

**Берлин, октябрь 2026** — Сегодня команда Bobr объявляет о публичном релизе Bobr 1.0 — open-source платформы (AGPL-3.0), которая объединяет управление требованиями, бэклогом и спецификациями в единую git-native систему, с которой напрямую работают AI-агенты: Claude Code, Cursor, Codex, Devin, Jules и другие.

**Проблема, которую решает Bobr.** AI-агенты уже умеют писать код, но они не знают *зачем*. Команды из 3–15 человек, использующие AI для разработки, сталкиваются с одним и тем же: контекст разбросан между Notion, Slack, Jira и головой тимлида. Каждый разработчик настраивает свой workflow. PM не видит, что делают агенты. А когда кто-то уходит из команды, workflow уходит вместе с ним.

**Что такое Bobr.** Bobr хранит всё, что нужно для разработки, в одном месте — директории `.bobr/` внутри git-репозитория проекта:

- **Требования** — Vision & Scope, Use Cases по методологии Вигерса
- **Бэклог** — задачи, баги, идеи с AI-assisted приоритизацией
- **Спецификации** — proposal → design → tasks для каждого изменения
- **Knowledge Base** — протоколы встреч, документы клиентов, решения
- **Трассировка** — от бизнес-потребности до конкретного PR

**Ключевое отличие: Bobr — инструмент для AI, а не AI-инструмент.** Bobr не содержит встроенного AI. Вместо этого Bobr предоставляет три уровня интеграции для любого AI-агента:

1. **Файлы** — агент читает `.bobr/` директорию как обычные файлы
2. **CLI** — агент вызывает `bobr backlog add --output json`
3. **MCP Server** — агент подключает Bobr как tool provider по протоколу MCP

Это значит, что Bobr работает с *любым* AI-агентом — текущим и будущим — без привязки к конкретному провайдеру.

**Результаты ранних пользователей:**

- **Команда Expecto** (5 человек, B2B fintech): cycle time от идеи до production сократился с 3 дней до 4 часов. 117 спецификаций, 64 завершённых change, 100% кода написано AI-агентами — с полным контекстом из Bobr.
- **Студия разработки из Мюнхена** (12 человек, 4 проекта): "Раньше при смене проекта разработчик тратил день на onboarding. Теперь Bobr генерирует CLAUDE.md с полным контекстом — 15 минут, и агент уже работает."
- **Solo-разработчик, SaaS**: "Я использовал Taskmaster AI, потом перерос. Bobr дал мне требования, спецификации и бэклог в одном формате. Claude Code читает `.bobr/` и понимает не только *что* делать, но *почему*."

**"Built with itself".** Bobr — первая платформа управления разработкой, которая была полностью построена с помощью себя самой. Начиная с Phase 0, команда вела бэклог Bobr в Bobr, писала спецификации в `.bobr/specs/`, а AI-агенты получали контекст из `.bobr/` для написания кода Bobr.

Bobr 1.0 доступен на [github.com/usebobr/Bobr](https://github.com/usebobr/Bobr). Облачная версия Bobr Cloud с командными функциями выходит в закрытую бета.

## INTERNAL FAQ

### Для пользователей

**Q: Чем Bobr отличается от Jira / Linear?**
A: Jira и Linear — традиционные трекеры, в которых AI-агент — надстройка. Bobr спроектирован с нуля как инструмент, с которым AI-агенты работают напрямую. Данные хранятся в git (не в проприетарной БД), формат — машиночитаемый Markdown+YAML, и три уровня интеграции работают с любым агентом без специальных плагинов.

**Q: Чем Bobr отличается от Taskmaster AI / Backlog.md?**
A: Taskmaster AI и Backlog.md — отличные инструменты для solo-разработчика. Bobr добавляет то, чего у них нет: управление требованиями (по Вигерсу), Knowledge Base, spec-driven development (proposal → design → tasks), полную трассировку, и командные функции. Bobr — следующий шаг, когда вы переросли git-native backlog для одного человека.

**Q: Чем Bobr отличается от Hamster?**
A: Hamster — SaaS для командного alignment (briefs → tasks). Bobr — open-source, git-native, и покрывает весь цикл: от требований до deployment. Hamster не хранит данные в git и не имеет management требований. Bobr можно self-host и он не привязан к конкретному AI-провайдеру.

**Q: Чем Bobr отличается от Kiro (Amazon)?**
A: Kiro — IDE со spec-driven workflow (requirements → design → tasks). Bobr — не IDE, а платформа данных и инструментов, с которой работает *любой* IDE/агент. Bobr покрывает также requirements management, Knowledge Base и бэклог — то, чего нет в Kiro. Kiro и Bobr могут работать вместе: Kiro как agent, Bobr как source of truth.

**Q: Нужно ли менять мой текущий IDE / агент?**
A: Нет. Bobr работает с тем, чем вы уже пользуетесь. Claude Code, Cursor, Codex, Copilot, Devin, Jules — все могут читать `.bobr/` файлы (L1), вызывать CLI (L2), или подключать MCP server (L3). Bobr — tool provider, не замена вашему агенту.

**Q: Что если я не использую AI-агентов?**
A: Bobr полезен и без AI. Это structured backlog + spec-driven development + requirements management в git. Человек может делать всё то же самое: создавать задачи, писать спецификации, отслеживать требования. AI-агенты — бонус, не обязательное условие.

**Q: Как Bobr хранит данные?**
A: Артефакты (backlog items, specs, requirements, changes) — в директории `.bobr/` внутри вашего git-репозитория. Это Markdown-файлы с YAML-метаданными. Они версионируются вместе с кодом, работают offline, и не привязаны к внешнему сервису. Для командных функций (search, analytics, roles) используется PostgreSQL, но `.bobr/` остаётся source of truth.

**Q: Это бесплатно?**
A: Bobr Core — open-source (AGPL-3.0), бесплатный навсегда. Bobr Cloud (hosted, team features, SLA) — по подписке.

### Для команды / технические

**Q: Почему Bobr не содержит встроенного AI?**
A: Три причины. (1) AI-агенты меняются быстрее чем любой продукт — привязка к конкретному API означает постоянную гонку. (2) Пользователи уже имеют своего агента (Claude Code, Cursor, Codex) — им не нужен ещё один. (3) Zero AI dependency = проще, дешевле, предсказуемее, безопаснее. Bobr решает другую задачу: structured data + tools для агентов.

**Q: Как Bobr зарабатывает?**
A: Модель GitLab/Grafana: open-source core (AGPL-3.0) + коммерческая облачная версия. Bobr Cloud включает: hosted MCP server, team management, RBAC, audit log, analytics dashboard, Knowledge Base с семантическим поиском, SSO. Self-hosted enterprise — по коммерческой лицензии.

**Q: Почему AGPL, а не MIT?**
A: AGPL защищает от ситуации, когда крупный вендор берёт код и создаёт конкурирующий hosted-сервис без вклада обратно в community. MIT не даёт такой защиты (Height.app закрылся, их код мог бы быть просто скопирован). AGPL используют Plane (46K stars), GitLab, Grafana — это proven модель.

**Q: Каков главный риск?**
A: Что мы построим слишком сложный инструмент, который отпугнёт пользователей. Height.app — proof этого риска. Митигация: dogfooding (мы сами — первые пользователи), progressive disclosure ("just add a task" как entry point), CLI-first (zero-overhead для разработчиков).

**Q: Почему git-native, а не чисто cloud DB?**
A: (1) AI-агенты уже умеют читать файлы — это L1 интеграция "бесплатно". (2) Данные версионируются с кодом — diff, blame, history без дополнительных инструментов. (3) Работает offline. (4) Нет vendor lock-in — если Bobr исчезнет, данные остаются в вашем репозитории. (5) Портабельность — переход на другой инструмент = чтение Markdown-файлов.

**Q: Не получится ли конфликт "ещё один инструмент"?**
A: Bobr заменяет, а не добавляет. Вместо Notion (для требований) + Jira (для бэклога) + Confluence (для specs) + отдельной настройки CLAUDE.md — один `.bobr/` в репозитории. Меньше инструментов, не больше.

Единственное, чего не хватает — визуального drag-and-drop для бэклога в web UI. Но CLI + MCP покрывают 90% потребностей, так что это not a blocker.

Если вы команда из 3–15 человек и AI пишет ваш код — попробуйте Bobr. Это не *ещё один трекер*. Это недостающий слой между вашими требованиями и вашими агентами."

## PRE-MORTEM: "Bobr провалился. Почему?"

> Упражнение: представим, что через год Bobr не взлетел. Анализируем возможные причины.

### Сценарий 1: "Слишком сложно для начала"
**Что случилось**: Пользователи пришли, увидели `.bobr/` с 6 поддиректориями, Vision & Scope, Use Cases, Change workflow из 7 шагов — и ушли обратно к Taskmaster AI, где PRD → tasks за одну команду.

**Урок**: Progressive disclosure критичен. Entry point должен быть: `bobr init && bobr backlog add "Fix login bug"`. Не заставлять писать Vision & Scope ради одного баг-фикса.

**Митигация в PRD**: Phase 0 = только backlog. Requirements и specs — Phase 1+. "Just add a task" как entry point.

### Сценарий 2: "Height.app 2.0 — слишком рано для рынка"
**Что случилось**: Рынок всё ещё фрагментирован. Команды не готовы переходить на новый инструмент. Jira + AI plugin достаточно для большинства. Height.app закрылся по этой же причине.

**Урок**: Не пытаться заменить Jira для 1000-человечных команд. Фокус на 3–15 человек, уже использующих AI-агентов, уже чувствующих боль.

**Митигация**: ICP жёстко определён. Dogfooding на Expecto. Не идти в enterprise до product-market fit.

### Сценарий 3: "Агенты научились сами"
**Что случилось**: Claude Code 5.0 / Codex 3.0 получили встроенное управление задачами, spec-driven workflow, knowledge base. Bobr стал не нужен.

**Урок**: Если крупный вендор решит эту задачу — мы проиграем. Но вендоры делают инструменты для *своего* агента. Bobr — agent-agnostic.

**Митигация**: P8 (Tool Provider) — ключевой принцип. Bobr ценен именно тем, что работает с *любым* агентом. Пока есть 2+ конкурирующих агента — есть рынок для Bobr.

### Сценарий 4: "Open source не взлетел"
**Что случилось**: GitHub stars не набрались. Нет community. Нет контрибьюторов. Проект остался internal tool.

**Урок**: Open source — это marketing + community + documentation. Недостаточно выложить код.

**Митигация**: README с compelling story. Quick start за 2 минуты. "Built with itself" как social proof. Статьи о methodology (Wiegers + AI). Интеграция с Taskmaster AI ecosystem.

### Сценарий 5: "Dogfooding создал bias"
**Что случилось**: Bobr идеально подходит для Expecto-подобных проектов (FastAPI + React, B2B), но не работает для мобильной разработки, ML-пайплайнов, embedded.

**Урок**: N=1 пользователь опасен. Нужны разные use cases рано.

**Митигация**: Привлечь 2–3 beta-команды из разных доменов к Phase 3. Не generalize из одного проекта.
